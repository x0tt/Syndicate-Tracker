# 🏆 Syndicate Tracker (Xanderdu)

A full-stack betting syndicate management suite combining a highly visual Streamlit dashboard with an always-on Telegram background worker, powered by Google's Gemini AI and LangChain.

The system automatically syncs with a Google Sheets ledger, auto-grades pending matches using The Odds API, and features an AI "Betbot" capable of answering complex SQL queries about betting performance in natural language. A built-in eval harness measures agent accuracy and catches regressions whenever the prompt or schema changes.

---

## ✨ Core Features

* **📊 Mobile-Optimised Dashboard:** Streamlit UI featuring Plotly visualisations including Bankroll Splines, Flow of Money Sankeys, Odds Beeswarms, Violin plots, and Member Radar charts. Supports a `?view=sankey` presentation mode for fullscreen display.
* **🤖 Betbot (LangChain SQL Agent):** Gemini-powered agent that queries the local SQLite database to answer natural language questions about syndicate performance. Secured by a two-layer guardrail: a prompt-level read-only instruction and a pre-flight `is_safe_query()` check in the bot runner.
* **⚙️ Automated Grading Engine:** Background polling automatically grades pending bets as Wins, Losses, or Pushes using live data from The Odds API. Results are written back to Google Sheets with retry logic and a structured failure log for guaranteed delivery.
* **📝 The Chronicler:** Weekly scheduled report (Wednesdays at 09:00) generated by Gemini and broadcast to the syndicate Telegram group. Rotates through 9 distinct personas — The Statistician, The Pirate, The Pundit, and others.
* **📱 Telegram Bot:** Long-polling bot that routes messages, handles on-demand commands (`? pending`, `? bank`, `? leaderboard`, `? streaks`, `? report`, `? status`), fetches upcoming fixtures, and delivers scheduled reports. Replies to whichever chat the message originated from.
* **🧪 Eval Harness:** Automated test suite that runs 39 gold-standard questions against the live SQL agent and scores the responses. Catches regressions when the prompt or schema changes. Expected values auto-refresh from the live ledger CSV on every bot startup.

## 🏗️ Architecture & Data Flow

```
Google Sheets (source of truth)
        │
        ▼
syndicate_ledger.csv  ◄──── sync_local_csv() on startup + before grading
        │
        ▼
ledger.db (SQLite)  ◄────── build_database() rebuilds views on every startup
        │
   ┌────┴────┐
   ▼         ▼
app.py    bot_runner.py
(Streamlit)  (background worker)
   │         │
   └────┬────┘
        ▼
   LangChain SQL Agent (Betbot)
        │
        ▼
   Gemini API
```

Key design decisions:
- **Views not raw tables:** Eight pre-aggregated SQLite views (`v_summary`, `v_by_bet_type`, `v_by_user`, etc.) are rebuilt on every startup. All views filter `status IN ('Win','Loss','Push') AND stake > 0` to precisely match the Python KPI definitions — preventing deposit rows and free bets from corrupting ROI figures.
- **Structured failure log:** Write failures to Google Sheets are logged as replayable JSON records in `logs/failed_writes.log` and automatically retried on the next bot startup via `replay_failed_writes()`.
- **Eval-driven development:** Before changing any prompt or schema, run `run_evals.bat` to get a baseline score. Run again after the change. Any regression is caught by name.

## 🚀 Installation & Setup

### 1. Prerequisites
* Python 3.10+
* Google Cloud Service Account credentials (`credentials.json`) with Editor access to the Google Sheet
* API keys: Gemini (Paid tier 1 recommended), Telegram Bot, The Odds API

### 2. Clone & Install
```bash
git clone https://github.com/yourusername/syndicate-tracker.git
cd syndicate-tracker
python -m venv .venv
.venv\Scripts\activate      # Windows
# source .venv/bin/activate  # Mac/Linux
pip install -r requirements.txt
```

### 3. Environment Configuration (`.env`)
Create a `.env` file in the project root:

```env
# APIs
GEMINI_API_KEY=your_gemini_api_key
GEMINI_MODEL=gemini-2.0-flash-lite
ODDS_API_KEY=your_odds_api_key
TELEGRAM_BOT_TOKEN=your_telegram_bot_token

# Telegram Routing
TELEGRAM_CHAT_ID=your_primary_group_chat_id
TEST_CHAT_ID=your_personal_dm_chat_id

# Google Sheets
GSHEET_ID=your_google_sheet_id
GSHEET_TAB=syndicate_ledger_v3
GOOGLE_CREDENTIALS_PATH=credentials.json

# Core App Settings
OPENING_BANK=0.00

# Feature Flags
TEST_MODE=false             # true = redirect scheduled messages to TEST_CHAT_ID
USE_GSHEETS_LIVE=true
USE_ODDS_API_LIVE=true
GRADING_DRY_RUN=false
BETBOT_LIVE=true
CHRONICLER_LIVE=true
```

### 4. Google Sheet Schema
The sheet tab must contain these exact column headers:

`uuid` · `date` · `user` · `home_team` · `away_team` · `competition` · `bet_type` · `selection` · `odds` · `stake` · `status` · `actual_winnings` · `matchday` · `sport`

### 5. Telegram User Mapping
Update `TELEGRAM_USER_MAP` in `bot_runner.py` with each member's Telegram user ID. Members can find their ID by messaging `@userinfobot` on Telegram.

### 6. Google Sheets Connectivity
If write-back isn't working, run the diagnostic tool before debugging anything manually:
```bash
python diagnose_sheets.py
```
It checks credentials, sheet access, tab existence, and performs a test write/delete.

## 🏃 Running the Ecosystem

Run both processes simultaneously in separate terminals:

```bash
# Terminal 1 — Web dashboard
streamlit run app.py

# Terminal 2 — Background worker (Telegram bot + grading scheduler)
python bot_runner.py
```

`bot_runner.py` performs on startup, in order: Google Sheets sync → CSV → SQLite rebuild → eval case refresh → agent initialisation → failed write replay → pending bet grading.

## 🧪 Running the Eval Suite

The eval harness verifies that the Betbot SQL agent answers correctly against the current ledger. Run it before and after any change to `agent.py`, `db.py`, or the prompt.

```bash
# Full suite (Windows)
run_evals.bat

# Tagged subset
run_evals.bat roi
run_evals.bat security
run_evals.bat members

# Or directly
python evals/run_evals.py
python evals/run_evals.py --tag markets
python evals/run_evals.py --output evals/results_before.json
```

Results are saved as timestamped JSON files in `evals/`. The `refresh_cases.py` script auto-updates expected values from the live CSV — run it manually after a large grading batch, or it runs automatically on every bot startup.

**Current baseline: 39/39** (as of March 2026 ledger snapshot).

## 📱 Telegram Bot Commands

All commands are prefixed with `?`, `betbot:`, or `bot:`.

| Command | Description |
|---|---|
| `? pending` | Lists all ungraded bets |
| `? bank` | Current bankroll and total P/L |
| `? leaderboard` | Member rankings by profit |
| `? streaks` | Current win/loss streaks per member |
| `? report` | Generates an on-demand Chronicler report |
| `? status` | Shows current mode (TEST/LIVE) and feature flags |
| `? help` | Command reference |
| `? <any question>` | Natural language Betbot query (e.g. `? what's our ROI on BTTS bets?`) |

## 🔒 Security Notes

- The SQL agent operates on a standard SQLite connection. DML instructions are blocked at two layers: the system prompt explicitly forbids INSERT/UPDATE/DELETE/DROP, and `bot_runner.is_safe_query()` runs a pre-flight Gemini check on every Telegram query before it reaches the agent.
- `.env` and `credentials.json` are excluded from version control via `.gitignore`. Never commit these files.
- `logs/failed_writes.log` may contain bet data — treat it as sensitive and exclude from version control.
